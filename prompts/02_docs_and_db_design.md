### [Prompt 2] Prisma 모델링 및 10,000건 시딩 (대용량 처리) 전략 수립

**(구현 계획이 승인된 후, AI 에이전트가 자체적으로 판단한 실행 제안 및 코드 작성 내용)**

**AI 판단의 핵심 내용 (`prisma/seed.ts` 작성 시 고려사항):**
- **요구사항**: 서버를 띄울 때 "1분 이내"에 학과 10개, 교수 100명, 학생 1만 명, 강좌 500개를 넣어야 함.
- **결정 사항**:
  1. 루프 안에서 개별 `create`를 태우면 늦어지므로, `Prisma.createMany()`를 활용한 배치 인서트로 속도 증가.
  2. 강좌 생성 시 교수의 일정이 겹치지 않게 하기 위해 Set 레이어로 시간 슬롯 추적.
  3. 학생 및 교수 이름의 랜덤화를 위해 "성+가운데글자+끝글자" 배열 로직 도입 (현실적인 데이터 덤프).

### [Prompt 3] SQLite WAL 모드 에러 트러블슈팅

**(개발 중 발생한 SQLite PRAGMA 에러에 대한 해결 과정)**

**발생 상황**:
```
PrismaClientKnownRequestError: Invalid `prisma.$executeRaw()` invocation: Execute returned results, which is not allowed in SQLite.
```

**AI의 원인 파악 및 해결**:
- `PRAGMA journal_mode=WAL;`은 DB 환경을 WAL 모드로 바꾸어 동시성 트랜잭션의 `BUSY` 락을 줄이는 핵심 설정임.
- 그러나 SQLite의 PRAGMA 문은 `SELECT` 처럼 결과 셋을 반환함.
- **직접적인 해결책**: Prisma에서 `$executeRaw` 대신 결과를 반환받아 무시할 수 있는 `$queryRawUnsafe("PRAGMA journal_mode=WAL;")`로 변경하여 에러없이 1만 건 데이터를 0.3초 만에 넣는 것에 성공.

---

## 설계 시 얻은 인사이트 (Codify)
1. **SQLite와 동시성**: SQLite는 단일 파일 DB라 동시 쓰기에 취약하지만, `WAL` 저널 모드를 켜면 읽기와 쓰기 동시 처리가 상당히 개선됨을 확인함. 평가를 위한 경량화 환경 구성 시 필수적인 패턴임.
2. **배치 인서트**: Prisma `createMany`와 함께 청크 단위 처리를 할 경우 1만 건 데이터를 1초 이하로 단축시킬 수 있었음.
