## 1. 기획 요청 분석

### 1.1 명시된 요구사항

| 기능 | 설명 |
|------|------|
| 학생 목록 조회 | 전체 학생 리스트 반환 |
| 강좌 목록 조회 | 전체/학과별, 정원/현재 인원/시간 포함 |
| 교수 목록 조회 | 전체 교수 리스트 반환 |
| 수강신청 | 정원/학점/시간 충돌 검증 후 등록 |
| 수강취소 | 수강 내역 삭제 |
| 내 시간표 조회 | 특정 학생의 이번 학기 수강 강좌 목록 |
| 헬스체크 | `GET /health` → HTTP 200 |

### 1.2 핵심 비즈니스 규칙 (명시됨)

- 정원 초과 절대 불가 (동시 요청 시에도 1명만 성공)
- 학생 1인당 최대 **18학점** 신청 가능
- 같은 시간대에 두 강좌 동시 수강 불가 (시간 충돌)

## 2. 명시되지 않은 요구사항 — 판단 및 결정

### 2.1 인증/인가

**결정**: 인증 없음 (Student ID를 요청 파라미터로 직접 전달)

**근거**:
- 화면은 프론트팀 담당이며, 서버 측 평가에 집중
- 인증 구현은 핵심 기능(동시성 제어) 구현 시간을 빼앗음
- 평가자 테스트 편의성: 학생 ID를 직접 지정해 바로 API 호출 가능

### 2.2 수강 취소 정책

**결정**: 제한 없음 — 언제든지 취소 가능

**근거**:
- 명시적인 취소 기한이 없으므로 가장 단순한 정책 선택
- 취소 즉시 `enrolled` 카운트 감소 및 정원 복원

### 2.3 재수강 허용 여부

**결정**: 동일 강좌 중복 신청 불가 (이미 신청한 강좌는 재신청 불가)

**근거**: 동일 강좌를 두 번 수강하는 것은 현실적으로 불합리. 409 Conflict 반환.

### 2.4 선수과목, 학년별 제한

**결정**: 선수과목 및 학년 제한 없음

**근거**: 문서에 명시되지 않았으며, 구현 복잡도 대비 평가 우선순위 낮음

### 2.5 시간표 슬롯 정의

**결정**: 요일 + 시작시간 + 종료시간 형태로 정의

```
예시: 월요일 09:00 ~ 10:30
포맷: { day: "MON", startTime: "09:00", endTime: "10:30" }
```

**요일 코드**: `MON`, `TUE`, `WED`, `THU`, `FRI`
**시간 범위**: 09:00 ~ 21:00, 1시간 30분 단위 강의 슬롯

**시간 충돌 판단**: 두 강의 시간이 겹치는 경우(endTime > startTime of other, same day) 충돌로 처리

### 2.6 학점 분포

**결정**: 강좌당 2~3학점 (랜덤 배정)

| 학점 | 비율 |
|------|------|
| 2학점 | 30% |
| 3학점 | 70% |

### 2.7 강좌별 정원

**결정**: 20~50명 랜덤 배정

### 2.8 학기 정의

**결정**: 현재 학기는 `2026년 1학기`로 고정 (시스템 내 상수로 관리)

## 4. 동시성 제어 전략

### 4.1 핵심 문제

> 정원이 1명 남은 강좌에 100명이 동시에 신청할 때, 정확히 1명만 성공해야 한다.

### 4.2 선택한 전략: Prisma 인터랙티브 트랜잭션 + 조건부 업데이트

**방법**: 수강신청 `POST /api/enrollments` 호출 시 아래 순서를 **하나의 원자적 트랜잭션**으로 처리

```
1. 트랜잭션 시작
2. 해당 강좌의 현재 enrolled 수를 조회 (SELECT ... FOR READ)
3. enrolled < capacity 조건 확인
4. 조건 충족 시 → Enrollment 레코드 INSERT + enrolled 카운트 +1 (UPDATE)
5. 조건 미충족 시 → 트랜잭션 롤백, 409/400 에러 반환
6. 트랜잭션 커밋
```

**SQLite WAL(Write-Ahead Logging) 모드 활성화**:
```sql
PRAGMA journal_mode=WAL;
```
- WAL 모드: 읽기-쓰기 동시 처리 성능 향상, 충돌 시 SQLITE_BUSY 오류 발생 최소화

### 4.3 한계 및 보완 방안

| 한계 | 보완 |
|------|------|
| SQLite는 단일 프로세스 환경에 최적화됨 | 단일 Next.js 서버 프로세스 내에서 운용하므로 충분 |
| 극도로 높은 동시 요청 시 SQLITE_BUSY 발생 가능 | Prisma 재시도 로직(retry) 또는 큐(Queue) 방식 검토 |

> **실제 프로덕션 환경이라면**: PostgreSQL의 `SELECT ... FOR UPDATE` 또는 Redis 기반 분산 락을 사용하는 것이 권장됩니다. 이 내용은 `docs/REQUIREMENTS.md`에 명시하며 평가자에게 알립니다.

## 5. 데이터 생성 전략

### 5.1 규모

| 항목 | 수량 |
|------|------|
| 학과 | 10개 |
| 교수 | 100명 |
| 학생 | 10,000명 |
| 강좌 | 500개 |

### 5.2 현실적인 데이터 생성 방법

- **이름**: 한국어 성(30개) × 이름 글자(30×30) 조합으로 약 27,000가지 경우의 수 → 충분
- **학과명**: 컴퓨터공학과, 전자공학과, 경영학과 등 10개 미리 정의
- **강좌명**: 자료구조, 알고리즘, 운영체제 등 학과별 현실적인 과목명 풀(pool)에서 랜덤 조합
- **시간표**: 요일(5개) × 시간 슬롯(8개) 조합 중 랜덤 배정, 교수당 시간 충돌 없도록 관리

### 5.3 속도 목표

목표: **1분 이내** 완료
전략:
- Prisma `createMany()` 배치 삽입 사용 (개별 INSERT 대비 10배 이상 빠름)
- 학생 10,000명을 500명씩 20개 배치로 나누어 삽입
